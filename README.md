# TaskFlow — REST API Система управления проектами и задачами

**TaskFlow** — это учебный корпоративный таск-трекер (аналог Jira или ClickUp), разработанный на базе экосистемы **Spring Boot 4**. Проект спроектирован как полноценное REST API приложение для организации совместной работы команд: создания изолированных проектов, распределения задач внутри них и гибкого разграничения прав доступа.

Цель проекта — демонстрация практического применения ключевых модулей Spring Framework (Core, Boot, Data JPA) и принципов чистой архитектуры.

## Стек технологий
* **Язык разработки:** Java 21
* **Фреймворк:** Spring Boot 4.0.7
  * Spring Core (IoC/DI)
  * Spring Web (REST API, Глобальная обработка ошибок)
  * Spring Data JPA (Работа с БД, Hibernate)
  * Spring Boot Validation (Валидация входящих данных)
  * Swagger (онлайн документация, ui для HTTP-запросов)
* **База данных:** PostgreSQL
* **Инструменты:** Maven

## Архитектура приложения
1. **Слой контроллеров:** Принимает HTTP-запросы в формате JSON, валидирует входящие DTO и передает данные дальше
2. **Слой бизнес-логики:** Чистая бизнес-логика приложения, управление транзакциями и валидация бизнес-правил
3. **Слой доступа к данным:** Интерфейсы Spring Data JPA для безопасного и удобного взаимодействия с базой данных
4. **Доменная модель:** JPA-сущности, описывающие структуру таблиц в БД и связи между ними. DTO для использования слоем контроллеров

## Ключевая бизнес-логика и правила

### 1. Жизненный цикл задачи (Task Workflow)
Каждая задача строго следует рабочему процессу, что исключает хаос в системе:
* Статусы задачи: `TODO` (В планах), `IN_PROGRESS` (В работе), `DONE` (Выполнено)
* **Бизнес-правило:** Все новые задачи создаются автоматически в статусе `TODO`. Прямой переход из `TODO` в `DONE` запрещен. Задача обязана пройти промежуточную стадию `IN_PROGRESS`

### 2. Изоляция данных
* При удалении проекта срабатывает каскадное удаление — все связанные с ним задачи автоматически стираются из базы данных

### 3. Ресурсы и валидация
* В системе настроена жесткая валидация: email пользователей проверяется на соответствие международному формату, а названия задач и проектов не могут быть пустыми или состоять из одних пробелов
* Нельзя зарегестрировать двух пользователей на 1 email

## Ролевая модель
Доступ к эндпоинтам разграничен на основе ролей пользователей:
* **`USER`**: Может просматривать и редактировать только те задачи, в которых он является создателем или участником
* **`ADMIN`**: Обладает глобальным доступом. Может удалять любые проекты/задачи

## Структура базы данных
* **User** `1 —— M` **Project** (Пользователь является создателем многих проектов)
* **Project** `1 —— M` **Task** (В одном проекте может быть много задач)
* **User** `1 —— M` **Task** (Пользователь может быть исполнителем многих задач)

## Как запустить проект локально

### Шаги для запуска
1. Клонируйте репозиторий с проектом:
   ```bash
   git clone https://github.com
   cd taskflow
   ```
2. Настройте параметры подключения к вашей базе данных в файле `src/main/resources/application.yml` (укажите ваш url, username и password)
3. Используйте docker compose
4. Приложение запустится локально на порту `8080`. Swagger: `http://localhost:8080/swagger-ui/index.html`
